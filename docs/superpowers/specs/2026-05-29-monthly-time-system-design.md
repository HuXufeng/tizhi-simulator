# 体制内模拟器 - 时间系统改造设计（按天→按月）

**文档版本**: 1.0
**创建日期**: 2026-05-29
**状态**: 已批准

---

## 1. 动机

当前游戏以"天"为最小时间单位，每天包含上午/下午/晚上共10个时间槽，玩家需要频繁选择任务，操作量大但策略深度不足。改为"月"后，精简操作频次，让每次选择更有分量，生涯推进感更强。

## 2. 时间结构

### 2.1 改前 vs 改后

| 维度 | 改前（按天） | 改后（按月） |
|------|-------------|-------------|
| 主单位 | day（天） | month（月） |
| 子单位 | timePart（时段）+ timeSlot（槽） | phase（阶段） |
| 每轮选择次数 | 10次/天 | 3次/月 |
| 定位 | 精细但繁琐的日程管理 | 精简的战略规划 |

### 2.2 月内阶段

| 阶段 | 对应原时段 | 说明 |
|------|-----------|------|
| 上旬 early | morning | 月初规划，适合重点工作 |
| 中旬 mid | afternoon | 常规推进 |
| 下旬 late | evening | 收尾冲刺，赶工 |

每个阶段玩家选择 **1 个行动**（任务），每月共 3 次行动。

### 2.3 生涯阶段对照

| 游戏阶段 | 月份范围 | 描述 |
|---------|---------|------|
| 新人期 | 第1-3月 | 熟悉环境 |
| 实习期 | 第4-12月 | 试用到转正完成 |
| 成长期 | 第13-60月 | 积累实力 |
| 骨干期 | 第61-120月 | 核心力量 |
| 老油条 | 第121-360月 | 等退休或冲刺 |
| 极限 | 第360月 | 游戏末期 |

## 3. 数据模型

### 3.1 gameState.js 改动

```javascript
// 旧
this.day = 1;
this.timePart = 'morning';
this.timeSlot = 0;

// 新
this.month = 1;
this.phase = 'early'; // early | mid | late
```

### 3.2 timeSystem.js 改造

**移除**:
- `timeParts`、`slotsPerPart`、`slotDuration`
- `advanceSlot()` 方法
- `getEfficiency()` 方法（不再需要，改为行动基础奖励）
- `_weekdayDescs`（周描述不再适用）
- `_morningDescs`、`_afternoonDescs`、`_eveningDescs`（改为月度描述）

**新增**:
- `advancePhase()` - 推进阶段，月末触发结算
- `getMonthPhaseName()` - 获取阶段中文名
- `_monthDescs` - 月度环境描述
- `_seasonDescs` - 四季描述（春季/夏季/秋季/冬季）

## 4. 游戏循环

### 4.1 月度循环

```
第1月 → 上旬（选任务）→ 执行 → 中旬（选任务）→ 执行 → 下旬（选任务）→ 执行 → 月末结算 → 第2月 → ...
```

### 4.2 月末结算

在每个月的**下旬**结束后触发：
- 发工资（每月固定，不再每30天一次）
- 月度支出扣除（房租、餐饮等）
- 需求变化（精力恢复、心态变化等）
- 关系值变化
- 事件检查
- 成就检查

### 4.3 初始流程

```
选择叙述者 → 序幕（同一天入职） → 第1月上旬开始（选第一个任务）
```

序幕内容保持不变，序幕结束后自动进入第1个月上旬。

## 6. 数值调整

| 数值 | 改前（按天） | 改后（按月） |
|------|-------------|-------------|
| 初始金钱 | 3000元 | 3000元 |
| 月薪 | 3000元/月 | 3000元/月（每月末发放） |
| 精力恢复 | 每天+20 | 每月自动恢复至100% |
| 任务基础奖励 | 每小时计算 | 每阶段固定值 |
| 游戏时长 | ~360天 | ~120月 |
| 事件触发频率 | 随机每时段 | 随机每月 |
| 晋升周期 | 约90天 | 约12月 |

## 7. 兼容与过渡

- 存档格式升级：`day` → `month`，旧存档兼容性不做保证
- 已完成任务记录：`day` 字段改为 `month`
- 叙述者冷却逻辑：基于月份重新计算间隔
